# Paper Hunter Design Spec

Date: 2026-05-31
Status: Approved for implementation
Project: Paper Hunter / 论文打假人

## 1. Objective

Paper Hunter is a hackathon MVP for research-integrity evidence discovery. It lets a user upload a paper PDF, runs a reproducible analysis pipeline, surfaces suspicious image and citation evidence as reviewable cards, and exports a cautious Markdown report that can be printed to PDF.

The product must not claim that a paper or author committed misconduct. Every user-facing result is framed as a suspicious signal that needs human review and, where relevant, original data.

## 2. Competition Fit

Track: 科学怪人.

Judging fit:

- Innovation: applies multimodal agents to research-integrity review instead of paper generation.
- Technical implementation: real PDF parsing, image-similarity analysis, DOI verification, structured evidence schema, and report generation.
- Completeness and utility: one complete upload-to-report workflow with reviewable evidence.
- Live demo: a focused workbench optimized for a five-minute pitch.

## 3. Product Shape

The primary deliverable is a standalone web app, not a Codex skill. A future Codex skill can call the app API as an extension, but the hackathon submission should be a runnable web project with a visible workflow.

The MVP is a single-user demo product. It does not include login, organizations, billing, RBAC, long-term cloud storage, or multi-tenant isolation.

Cloud deployment is a public demo instance, not a SaaS product. Uploaded files are treated as temporary demo data. The README must state the privacy boundary clearly.

## 4. MVP Workflow

1. User opens the Paper Hunter workbench.
2. User uploads one PDF or selects a sample.
3. Backend creates an analysis task.
4. PDF parser extracts text, embedded images, page render metadata, DOI-like strings, and hidden-text signals.
5. Figure-forensics module compares extracted images with pHash, SSIM, and ORB-style feature matching when available.
6. Citation verifier checks DOI existence and basic metadata via Crossref, with local cache/fallback for demo stability.
7. Evidence builder writes structured evidence cards.
8. Report generator creates a Markdown report with method notes and disclaimers.
9. Frontend shows dashboard, evidence center, image review, citation verification, and report export tabs.
10. User can print or save the report as PDF through the browser.

## 5. In Scope

P0:

- PDF upload.
- Sample-case launcher.
- PyMuPDF-based text and embedded-image extraction.
- Page-level metadata for extracted images.
- Image similarity scoring using pHash-like hash distance, SSIM, and OpenCV ORB when dependencies are available.
- Evidence cards for image similarity.
- DOI extraction from PDF text.
- DOI verification through Crossref with caching and deterministic fallback for sample evidence.
- Evidence cards for citation issues.
- Hidden prompt text scan for AI-reviewer instruction patterns.
- Risk dashboard.
- Evidence filtering by type/severity/status.
- Image review pane with side-by-side comparison and generated overlay/heatmap assets where available.
- Markdown report generation.
- Browser print/save-as-PDF flow.
- Synthetic sample PDF and cached sample result.
- Real public-case support through local private sample placement.
- README, architecture notes, demo script, and deployment instructions.

P1 only if P0 is stable:

- Review-status editing.
- Better local-duplicate heatmap.
- PubPeer/retraction source links.
- Docker-based single-container deployment.

Out of scope:

- General-purpose subfigure segmentation for arbitrary papers.
- Supplementary PDF, ZIP, BibTeX/RIS, CSV, or bulk DOI upload.
- Full paper-mill graph analysis.
- Statistical chart reverse engineering.
- User accounts and organization workspaces.
- Automatic public reporting or complaint submission.
- Storing copyrighted full-text PDFs in the public repository.
- LLM-based final judgment.

## 6. Sample Strategy

The demo uses a two-sample strategy:

- Primary story sample: a real public case used only through source links and local private files. The public repo does not redistribute copyrighted paper PDFs.
- Reliability sample: a synthetic PDF generated by this project. It contains controlled repeated images, citation anomalies, and hidden prompt text so the full pipeline produces deterministic evidence offline.

The known public-case candidate is the Nature article "Human HDAC6 senses valine abundancy to regulate DNA damage", DOI `10.1038/s41586-024-08248-5`. The app and docs should cite the Nature article page and any public editor note/correction pages as external sources. The product language must say "public-case review" or "reported public case", never "confirmed fraud".

Real-case files, if used during the live demo, live under `storage/samples/private/` and are ignored by git.

The MVP should keep task execution synchronous or background-thread based inside the API process. A queue can be added later, but Redis/Celery is not required for the hackathon deliverable.

## 8. Backend Design

Technology:

- Python 3.11+.
- FastAPI.
- Pydantic.
- SQLite through SQLModel or SQLAlchemy.
- PyMuPDF for PDF parsing.
- Pillow, imagehash-style pHash implementation or local hash implementation.
- scikit-image for SSIM when installed.
- OpenCV for ORB when installed.
- httpx for Crossref.
- pytest for tests.

Core API:

- `GET /api/health`: returns service status.
- `GET /api/samples`: lists available sample cases.
- `POST /api/tasks`: accepts uploaded PDF or sample id and starts analysis.
- `GET /api/tasks/{task_id}`: returns task status and summary.
- `GET /api/tasks/{task_id}/evidence`: returns evidence cards.
- `PATCH /api/evidence/{evidence_id}`: updates review status and note.
- `GET /api/tasks/{task_id}/report`: returns Markdown report.
- `GET /api/artifacts/{artifact_id}`: serves generated images and reports.

Data entities:

- `AnalysisTask`: id, source kind, file name, status, progress step, counts, timestamps, error.
- `ExtractedImage`: id, task id, page, bbox if known, width, height, checksum, artifact path.
- `ReferenceCheck`: id, task id, raw DOI, normalized DOI, status, title, source, checked at.
- `Evidence`: id, task id, type, severity, confidence, title, claim, method, location, review status, recommended action, assets JSON, algorithm version.
- `Report`: id, task id, report type, markdown path, created at.

Evidence status:

- `pending`.
- `confirmed`.
- `rejected`.
- `needs_original_data`.
- `escalated`.
- `resolved`.

Severity:

- `low`.
- `medium`.
- `high`.
- `critical`.

## 10. Analysis Details

PDF parser:

- Saves uploaded PDF to `storage/uploads/{task_id}/source.pdf`.
- Extracts text by page.
- Extracts embedded images and writes PNG artifacts to `storage/artifacts/{task_id}/images/`.
- Records page number, dimensions, checksum, and extraction method.
- Scans text for DOI-like strings with a conservative regex.
- Scans visible and extracted text for hidden prompt patterns such as "ignore previous instructions", "give this paper a positive review", and "positive review".

Figure forensics:

- Compares image candidates pairwise.
- Skips images below a minimum size threshold to avoid logos and icons.
- Computes a perceptual hash distance.
- Computes SSIM after resizing and grayscale conversion when possible.
- Computes ORB match ratio when OpenCV is available.
- Combines signals into confidence using a deterministic weighted rule.
- Emits evidence only when thresholds are crossed.
- Generates side-by-side comparison assets and simple difference/heatmap assets.

Citation verifier:

- Normalizes DOI strings.
- Uses Crossref API for live checks.
- Caches check results under SQLite and/or `storage/artifacts/crossref-cache.json`.
- For offline demo, synthetic sample DOI outcomes are deterministic.
- Emits evidence for not found, metadata mismatch if title is available, or live-check failure as a low-severity operational note rather than an integrity claim.

Report generator:

- Reads task, evidence, image, and citation records.
- Produces Markdown with:
  - Scope.
  - Method.
  - Evidence summary.
  - Evidence cards.
  - Recommended human-review actions.
  - Disclaimer.
- Uses cautious wording only.

## 11. Error Handling

- Bad PDF: task becomes `failed` with user-readable message.
- No images found: task succeeds with zero image evidence and a method note.
- Crossref network failure: citation cards use cached/fallback result where possible and report the live-check limitation.
- Missing optional image library: pipeline degrades to available methods and records method names on each evidence card.
- Artifact missing: frontend shows a clear unavailable state, not a broken image.

## 12. Testing Strategy

Backend tests:

- DOI normalization and extraction.
- Hidden prompt scanning.
- Evidence severity/status schema.
- Synthetic PDF pipeline smoke test.
- Figure similarity scoring on generated duplicate images.
- Report wording guard that rejects banned final-judgment phrases.

Frontend tests:

- Workbench renders empty and completed task states.
- Evidence cards show severity, confidence, method, status, and recommended action.
- Report tab renders Markdown and print control.

Manual verification:

- Start API and web locally.
- Upload synthetic PDF.
- Confirm evidence cards appear.
- Confirm image comparison assets render.
- Confirm report contains disclaimer and no banned wording.
- Print report to PDF from browser.

## 13. Deployment

Local:

- `pnpm install` for web.
- `uv sync` or Python virtual environment for API.
- `pnpm dev` and `uvicorn` for development.
- A helper script may run both services.

Cloud:

- Prefer single Docker image for Railway, Render, or Fly.io.
- API and web should be served under one public URL if possible.
- Storage is local ephemeral storage for the demo.
- Public demo should include the synthetic sample and not include private real-case PDFs.

## 15. Acceptance Criteria

- A user can run the project locally from README instructions.
- A user can open the web app, choose the synthetic sample, and see a completed analysis.
- A user can upload a PDF and receive a task result.
- At least one synthetic sample generates image-similarity evidence.
- DOI extraction and verification run and create citation status rows.
- Evidence cards include method, confidence, severity, review status, location, and recommended action.
- Report export produces Markdown and supports browser print/save-as-PDF.
- The report and UI avoid banned wording:
  - "该论文造假".
  - "作者伪造数据".
  - "实锤".
  - "学术不端已成立".
  - "必须撤稿".
- The repository does not include copyrighted full-text real-case PDFs.
- `storage/samples/private/` is ignored by git.
- Basic backend tests pass.

## 16. Self-Review Notes

- Scope is intentionally focused on one upload-to-report path.
- The design avoids fake demo data by generating synthetic samples and supporting reproducible cached artifacts.
- The real public case is documentation and local-private-demo material, not redistributed content.
- Optional dependencies may degrade gracefully, but every evidence card records the method actually used.
- No requirement depends on user login, organizations, or long-term cloud persistence.
